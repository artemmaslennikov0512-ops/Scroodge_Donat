# Выводы и рекомендации по MVP

Документ фиксирует сильные стороны проекта, выявленные пробелы и приоритизированный список задач для достижения рабочего MVP.

---

## Сильные стороны

- **Архитектура** — разделение на delivery / application / domain / infrastructure, порты для платежей и репозиториев.
- **Стек** — Next.js App Router, Prisma, Stripe, единый стиль и неплохой UX.
- **Покрытие сценариев** — регистрация, авторизация, создание доната (Stripe), цели (CRUD), верификация стримера (форма + загрузка файлов), админка верификаций.

---

## Соответствие выводов коду

| Вывод из ревью | Факт в коде |
|----------------|-------------|
| «Модели не согласованы, goalId в Donation» | В Prisma у `Donation` есть `goalId` и связь с `Goal`. Не согласованы: API создания доната и use case **не принимают** `goalId`, репозиторий не сохраняет его. |
| «Нет модели Subscription» | Подписок в схеме нет — запланировано в ROADMAP (п. 5). Для MVP можно не вводить. |
| «Нет реальной обработки платежей и webhook'ов» | **Частично неверно**: Stripe подключён (`stripe-gateway.ts`), webhook обрабатывает `payment_intent.succeeded/failed/canceled` и обновляет статус доната. Не хватает: обновления `Goal.currentAmount` при успешном платеже и опционально уведомления в Discord. |
| «Алерты не сохраняются» | **API есть**: `GET/PUT /api/alerts` с upsert по типам. В дашборде вкладка «Алерты» (`AlertsTab`) — заглушка, не загружает и не сохраняет настройки через API. |
| «Верификация: загрузка файлов, модерация» | Загрузка реализована: `POST /api/auth/streamer-verification` принимает `FormData`, `lib/upload.ts` (Vercel Blob). Админка: `/admin/verifications` и API верификаций есть. Нужно: убедиться, что `BLOB_READ_WRITE_TOKEN` задан в production. |
| «Нет проверки подписки при доступе» | Для MVP проверка подписки (платного тарифа) не реализована — при необходимости добавить после появления модели Subscription. |

## Что нужно доделать для MVP (приоритет)

### 1. API и данные

- [ ] **Список донатов стримера** — GET `/api/donations` (или `/api/dashboard/donations`) для текущего пользователя: только донаты по стримерам, привязанным к нему (через `Streamer.userId`). Подменить мок на дашборде `/dashboard/donations` реальным запросом.
- [ ] **Сохранение алертов** — во вкладке «Алерты» в дашборде: загрузка настроек через `GET /api/alerts`, форма редактирования, сохранение через `PUT /api/alerts`.
- [ ] **Цели и донаты** — CRUD целей уже есть (`/api/goals`, `/api/goals/[id]`). Нужно: при создании доната принимать опциональный `goalId`, сохранять в `Donation` и в webhook при `payment_intent.succeeded` увеличивать `Goal.currentAmount` и при достижении цели выставлять `isCompleted`.

### 2. Публичная страница доната и цели

- [ ] **Страница стримера** (`/streamer/[slug]`): показывать активные цели стримера (цели пользователя, у которого есть `Streamer` с данным `slug`).
- [ ] **Виджет доната** — опциональный параметр `goalId`. В форме доната выбор цели (или «Без цели»). Передавать `goalId` в `POST /api/donations/create`.
- [ ] **Цепочка goalId** — добавить `goalId` в `createDonationBodySchema`, `CreateDonationInput`, порт репозитория и `prismaDonationRepository.create`; в Stripe metadata можно передавать `goalId` для логов.

### 3. Верификация стримера

- [ ] Убедиться, что в production настроен Vercel Blob (`BLOB_READ_WRITE_TOKEN`) или аналог в `lib/upload.ts`.
- [ ] Админ-модерация уже есть; при необходимости добавить отклонение с комментарием и уведомление пользователю.

### 4. Платежи (Stripe)

- [ ] Уже есть: создание Payment Intent, редирект на success, webhook.
- [ ] Дополнительно: в `stripe-gateway.ts` в `handleWebhook` при `payment_intent.succeeded` по `metadata.goalId` обновлять `Goal.currentAmount` и при достижении — `isCompleted`.

### 5. Интеграция Discord

- [ ] При успешном донате (в webhook после обновления статуса) опционально отправлять уведомление в Discord (webhook URL в настройках стримера/пользователя или в `SystemSettings`). Переменная окружения типа `DISCORD_WEBHOOK_URL` или настройка в БД.
